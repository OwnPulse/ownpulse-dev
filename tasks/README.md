# Task System

Autonomous task queue for Claude Code sessions. Tasks are generated (e.g., from code/security reviews), stored in the workspace state directory, and picked up by sessions launched via `opdev`.

## Directory layout

```
ownpulse-dev/                              # Committed to git
  agents/task-runner.md                    # Agent definition with full protocol
  tasks/
    README.md                              # This file
    PROMPT.md                              # Bootstrap prompt for claude -p

<clone_root>/.ownpulse-dev/                # Runtime state (not in any repo)
  sessions.json                            # opdev session tracking
  tasks/
    index.json                             # Task list with priorities and worktree assignments
    <id>.md                                # Task plans (context, findings, step-by-step)
    <date>-<name>.json                     # Review data (raw output from review agents)
    <id>.lock                              # Claimed by a session (PID + timestamp)
    <id>.done                              # Completed (branch, summary, files changed)
```

Everything task-related — plans, reviews, locks, done markers — lives in one flat directory under `.ownpulse-dev/tasks/`.

## Configuration

The state directory is configured in `workspace.toml`:

```toml
[tasks]
# Base directory for all task state. Defaults to <clone_root>/.ownpulse-dev
state_dir = ".ownpulse-dev"
```

## Launching a session

```bash
# In any opdev session worktree:
claude -p "$(cat <opdev-repo>/tasks/PROMPT.md)"

# Or interactively — start claude and paste:
# Read <opdev-repo>/tasks/PROMPT.md and follow its instructions.
```

Replace `<opdev-repo>` with the path to your ownpulse-dev clone (e.g., `~/src/ownpulse/ownpulse-dev`).

The session will:
1. Read `.ownpulse-dev/tasks/index.json` to find the task list
2. Match its `$PWD` to a task, or pick the highest-priority unlocked one
3. Claim it by writing a `.lock` file
4. Read the `.md` plan and execute it
5. Commit changes and write a `.done` file

## Task lifecycle

```
available  →  locked  →  done
```

| State | Condition |
|-------|-----------|
| Available | `<id>.md` exists, no `.lock`, no `.done` |
| Locked | `<id>.lock` exists and PID is alive |
| Stale lock | `<id>.lock` exists but PID is dead — can be reclaimed |
| Done | `<id>.done` exists |

## Creating tasks

Tasks can be created manually or generated by review agents. Each task needs:

1. An `.md` plan file in `.ownpulse-dev/tasks/` with:
   - Metadata table (ID, repo, priority, related PRs, worktree dir)
   - Context section
   - Findings with file paths, line numbers, and fix descriptions
   - Step-by-step plan
   - Verification commands

2. An entry in `.ownpulse-dev/tasks/index.json`:
   ```json
   {
     "id": "my-task",
     "plan": "my-task.md",
     "repo": "ownpulse",
     "priority": "high",
     "worktree_dir": "/path/to/session/worktree"
   }
   ```

Priority levels: `critical` > `high` > `low`

## Reviews

Review data (raw JSON output from code-review or security-review agents) lives in the same `tasks/` directory, named `<date>-<name>.json`. Task plans reference these by filename for additional context.
